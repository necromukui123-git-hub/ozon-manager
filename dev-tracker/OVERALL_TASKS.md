# Ozon Manager 开发总任务

最后更新时间：2026-03-03  
负责人：团队 + Codex  
范围：把官方促销和店铺促销统一到一套交互，并让店铺促销在浏览器登录态下“低打扰执行”。

## 项目目标

让用户在同一套流程中操作官方促销和店铺促销，不需要理解二者的数据来源差异。

## 产品级要求

1. 前端保持一套交互模型，覆盖：
2. 活动列表
3. 报名
4. 退出
5. 亏损处理
6. 改价再推广
7. 活动商品同步
8. 店铺促销执行遵循“静默优先”：
9. 若 Ozon Seller 已登录，则静默执行。
10. 若未登录，仅在必要时打开登录页，登录后自动续跑。

## 架构决策（已锁定）

1. 官方促销：后端直连官方 API（保留现有链路）。
2. 店铺促销：浏览器插件执行（复用用户当前浏览器会话）。
3. 后端职责：编排、任务队列、鉴权、结果汇总。
4. 前端职责：统一展示与统一操作入口。

## 实施主线

### 主线 A：统一促销业务逻辑

1. 后端统一报名/退出。
2. 后端统一亏损处理/改价推广。
3. 支持一次操作混合官方活动 + 店铺活动。
4. 支持一次操作多个店铺活动。

### 主线 B：执行通道

1. 旧通道：automation agent（playwright/mock）继续可用。
2. 新通道：extension 任务接口 + 轮询/回报机制。
3. 插件需支持店铺活动同步、店铺报名/退出、remove-reprice-readd。

### 主线 C：前端集成

1. 保持一套操作流程。
2. 展示异步进度与结果。
3. 不向用户暴露执行引擎复杂度。

### 主线 D：交付与运维

1. 测试阶段通过“解压加载/zip 包”分发插件。
2. 后续迁移到 Chrome Web Store 分发。
3. 增加执行引擎路由控制与冲突防护。

## 未完成事项

1. 补齐 Chrome 商店上架清单与隐私文案（隐私政策、权限说明、素材与审核说明）。
2. 增加真实环境联调回归（多店铺 + extension/agent 混合在线 + 长时间轮询）。
3. 根据运行数据补充执行引擎路由监控指标（领取来源、fallback 次数、冲突阻断次数）。

## 本轮已完成（2026-03-02）

1. 已实现按店铺执行引擎模式：`auto` / `extension` / `agent`。
2. 已在后端任务领取链路中加入路由判断，`auto` 模式下默认 extension 优先，agent 仅在 extension 离线时领取。
3. 已实现 extension 任务分配归属校验（`assigned_agent_id` 与 `extension_id` 绑定）。
4. 已新增系统管理员插件状态接口与前端状态面板（在线/离线、最近任务、错误信息）。
5. 已完善非 localhost 自动同步策略：基于后端同源推导 + 可选管理端 Origin 白名单授权。
6. 已补充后端路由/授权关键逻辑测试（service 层）。

## 当前阶段完成标准

1. 官方 + 店铺在统一 UX 下可用。
2. 常规场景下店铺操作不出现可见自动化跳转。
3. 仅在必要时触发登录兜底。
4. 异步结果可按 job ID 持久化追踪。
